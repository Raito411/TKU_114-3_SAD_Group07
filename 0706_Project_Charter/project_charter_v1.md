# 任務 1：案例拆解

## 案例

社團活動報名與場地申請系統

## 小組名稱

Group07

## 利害關係人（Stakeholders）

請列出至少 5 個具體利害關係人，不可只寫「使用者」。

1. 社團成員：提出活動內容、申請所需資料。
2. 社團顧問/指導老師：對活動的內容提出意見、看法。
3. 場地管理員：管理各個社團間的活動場地確保彼此之間互不不衝突。
4. 申請系統管理員：對合法的場地申請進行註冊。
5. 外包承辦人士：針對社團人員提出的要求負責場地規劃。

## 痛點（Pain Points）

請列出至少 3 個目前流程中的問題。痛點需要描述目前問題，不是直接寫解法。

1. 活動報名者可能會取消報名，社團人員需要人工檢查空缺與通知候補者，可以新增遞補機制。
2. 每個人負責的內容會有不同步的情況發生，讓申請進度可以建立群組或定時開會，及時瞭解成員間的進度。
3. 若有要請校外人士幫忙佈置場地則可能需要替他們申請進出許可。

## 資料（Data）

請列出至少 5 種系統需要保存的資料。

1. 報名資料：姓名、電話等。
2. 場地申請核可證明：來自場地管理員。
3. 系統申請紀錄：來自申請系統管理員。
4. 遞補表單：若有缺席可以遞補。
5. 聊天記錄：確保彼此之間權益。

## 業務規則（Business Rules）

請列出至少 3 條業務規則或限制。

1. 若涉及金錢等重要事項需事先跟指導老師和校方報備、說明。
2. 若超過人數限制應把剩餘人數加進遞補名單。
3. 同一時段且同一場地只能有一個活動使用，避免衝突。

## 風險（Risks）

請列出至少 2 個可能造成系統失敗的風險。

1. 通知機制不可靠：使用者無法正確得知資訊，導致衝突和使用者感到困惑。
2. 申請系統說明不明確：讓學生需要一再準備資料，浪費時間。

## 最後選定主題：題目一(校內二手書面交預約系統)

## 選擇理由：

選擇「校內二手書面交預約系統」是因為此題目的使用者明確，主要包含買方、賣方與系統管理員，容易模擬實際使用情境，也方便訪談身邊有買賣二手書經驗的同學。此系統具有清楚的核心流程，例如賣方刊登書籍、買方搜尋與提出購買需求、雙方確認面交時間與更新交易狀態，符合本課程需要分析流程、資料與規則的要求。

此外，此題目需要管理的資料也很明確，包含書籍資料、使用者資料、交易紀錄、面交時間與交易狀態等，適合後續製作需求文件。相較於需要處理成績、教師審核或聊天內容統整的系統，二手書交易系統的範圍較容易控制，在 5 週內較能完成可展示的原型。本期可以聚焦在刊登、搜尋、預約面交與交易狀態追蹤，不處理線上付款、跨校交易與真實身分驗證，範圍較清楚且可行。

# 任務 4：撰寫問題陳述（problem statement）

使用句型完成問題陳述（problem statement）：

對於校內想購買二手書的學生、想出售二手書的學生而言，目前在校園二手書交易流程中，常發生書籍資訊分散在社群平台或聊天群組、交易狀態不明確、面交時間需要反覆私訊確認、買賣雙方難以追蹤交易進度等問題，可能造成學生找書效率低、重複詢問、交易延誤或交易取消。

本專案希望設計一個校內二手書面交預約系統，協助賣方集中刊登書籍資料，讓買方依書籍類別或關鍵字搜尋需要的教材，並協助買賣雙方確認面交時間與更新交易狀態。本期聚焦於書籍刊登、書籍搜尋、購買需求提出、面交預約與交易狀態追蹤，不處理線上付款、跨校交易、物流寄送與真實身分驗證。

# 任務 5：建立範圍陳述（scope statement）

## 專案目標（project goal）

讓校內學生間能夠方便買賣二手書。

## 範圍內項目（in-scope）

1. 買方可依據書籍類別、名稱等方式搜尋二手書。
2. 賣方可提供書籍照片、分類書籍類別、價格、名稱等資料。
3. 交易狀態可以即時顯示。
4. 可以讓買賣家預約地點與時間來交易。
5. 系統管理員要能取消不符合規範的買賣方資格。
6. 系統管理員要能處理、下架已完成的交易。

## 範圍外項目（out-of-scope）

1. 不開放跨校間的交易。
2. 不支援線上付款。
3. 不支援APP。
4. 不做自動估價。
5. 不對交易糾紛仲裁或賠償。

## 核心任務（core tasks）

請列出至少 3 個可展示任務。

1. 賣方新增一本二手書資料，包含書名、類別、價格、書況與照片。
2. 買方依書名或類別搜尋二手書，查看書籍詳細資料並提出購買需求。
3. 買賣雙方確認面交時間與地點，並將交易狀態更新為交易預約中或交易完成。

## 初步成功標準（initial success criteria）

1. 使用者可以搜尋和刊登書籍並預約地點與時間。
2. 系統可以區分交易的狀態，如待交易、交易預約中、交易完成。
3. 每筆書籍資料都能正確且清楚顯示書名、類別、價格、交易狀態。
4. 交易中預約的地點與時間僅能被買賣雙方及系統管理員讀取。
